==================================================
1. Mission
==================================================

Goal:
Reimplement a MATLAB-compatible language system as a modular project whose final build artifact can act as a MATLAB compiler and execution platform.

Primary objective:
- Compile and execute a large, practical subset of MATLAB source code.
- Preserve MATLAB-like language semantics where compatibility matters.
- Keep the architecture decomposed enough that many AIs or human contributors can work independently.

Core assumption:
- This project is not trying to clone every MATLAB toolbox on day one.
- The first serious deliverable is a compiler + runtime + standard library core.
- GUI-heavy desktop features, Simulink, App Designer, and niche toolboxes are later phases unless explicitly promoted.

Compatibility baseline for planning:
- Core MATLAB language
- Scripts
- Functions
- Anonymous functions
- Function handles
- Basic classdef
- Numeric arrays
- Logical arrays
- Char/string support
- Structs
- Cell arrays
- Indexing semantics
- Control flow
- File/module/package resolution
- Core math library

Final vision:
- A source-compatible compiler toolchain for MATLAB-like code
- An execution runtime
- A conformance suite
- A modular codebase that can continue evolving into broader MATLAB compatibility

==================================================
2. Non-Negotiable Architectural Rules
==================================================

- Every major subsystem must be independently testable.
- No giant "misc" modules.
- Every module must have:
  - a clear ownership boundary
  - a README or spec file
  - a public API boundary
  - unit tests
  - integration expectations
- The parser must not know runtime implementation details.
- The optimizer must operate on IR, not AST hacks.
- Runtime semantics must be centralized, not reimplemented ad hoc across compiler stages.
- Standard library compatibility must live above the runtime core.
- Experimental features must be isolated behind flags.
- Compatibility decisions must be written down in spec documents.
- If a behavior differs from MATLAB, the difference must be explicit and tested.

==================================================
3. Product Strategy
==================================================

The project will be built in layers:

1. Language front-end
2. Semantic analysis
3. Intermediate representations
4. Optimization pipeline
5. Runtime system
6. Standard library
7. Code generation and execution
8. Compatibility and packaging
9. Tooling and conformance infrastructure

The first usable system should be:
- headless
- command-line driven
- batch compile capable
- able to run nontrivial `.m` programs

Desktop IDE parity is not a phase-1 requirement.

==================================================
4. Explicit Non-Goals For Early Phases
==================================================

Do not block early progress on:
- full MATLAB desktop UX
- figure/window system parity
- Simulink
- GUIDE/App Designer parity
- GPU acceleration
- distributed computing
- full toolbox parity
- full MEX ABI compatibility
- JIT sophistication beyond the minimum needed for performance milestones

These may be designed for, but not required for phase-1 completion.

==================================================
9. IR Design Requirements
==================================================

HIR must preserve MATLAB-level semantics:
- matrix construction
- dynamic dispatch points
- workspace operations
- multiple returns
- exception and warning sites
- high-level indexing

MIR should lower:
- shape-normalized operations
- explicit allocation decisions
- explicit control flow
- builtin call forms

LIR should be:
- backend oriented
- explicit temporaries
- explicit memory ownership model
- easy to emit into bytecode, C, or LLVM

IR invariants checklist:

[ ] every IR level has a written invariant doc
[ ] every lowering pass documents preconditions
[ ] every pass has verifier logic
[ ] IR dump format is stable enough for regression tests
[ ] source locations survive enough stages for useful diagnostics
[ ] exception/error edges are explicit
[ ] multiple-result operations are first-class

Runtime subsystems:

5. String/char layer
6. Cell/struct/object layer
7. Function/method invocation layer
8. Error/warning/stack trace layer
9. File/path/environment layer
10. Reflection/introspection layer

[ ] tagged value or boxed object strategy chosen

[ ] shape/stride metadata policy defined
[ ] copy-on-write metadata defined

Memory model decisions:

[ ] reference counting vs tracing GC decision

Array engine capabilities:

11. Standard Library Strategy
==================================================

Split the standard library into tiers:

Tier 0: compiler/runtime intrinsics
- primitive arithmetic
- comparison
- allocation
- shape query
- indexing primitives

Tier 1: core language builtins
- `size`
- `length`
- `numel`
- `class`
- `isa`
- `zeros`
- `ones`
- `eye`
- `reshape`
- `permute`
- `sum`
- `prod`
- `min`
- `max`

Tier 2: high-frequency math and matrix library
- linear algebra basics
- decompositions needed for credibility
- trig/exponential/log functions
- random basics

Tier 3: compatibility surface expansion
- strings/text utilities
- struct/cell helpers
- filesystem utilities
- dates/times
- serialization

Rule:
- Builtins that affect semantics or performance-critical shape behavior belong as intrinsics.
- Pure convenience functions should live in library space when possible.

==================================================
12. Execution Modes
==================================================

The system should support more than one execution strategy.

Planned order:

1. Tree-walk or direct AST interpreter for bring-up only
2. Bytecode VM for stable early execution
3. AOT/native backend for serious compiler milestone
4. Optional JIT for performance phases

Why:
- Interpreter helps validate semantics fast
- Bytecode stabilizes execution semantics
- Native backend becomes the "compiler" story
- JIT is valuable later, but should not block phase-1 delivery

Execution mode checklist:

[ ] interpreter can run parser/semantic smoke tests
[ ] bytecode VM reaches conformance MVP
[ ] backend compiler can emit distributable artifact
[ ] runtime behavior is shared across all modes
[ ] differential tests compare interpreter vs compiled output

17. Recommended Module Ownership Model
==================================================

Each module should be owned like a mini-project.

For every major module create:
- `README.md`
- `SPEC.md`
- `TASKS.md`
- `DECISIONS.md`
- `TEST_PLAN.md`

Each module owner must answer:
- What does this module own?
- What is explicitly out of scope?
- What data structures are public?
- What invariants must other modules rely on?
- What diagnostics can it emit?
- What tests prove it works?

AI handoff rule:
- No AI should touch a module without first reading its `SPEC.md` and `TASKS.md`.

18. Multi-AI Collaboration Rules
==================================================

This project assumes multiple AIs may work over time.

Rules:

- Do not make silent semantic decisions.
- Record all behavior-affecting changes.
- Prefer narrow PRs or patches per module.
- Every new subsystem must include docs before deep implementation.
- If a module API changes, update downstream expectations immediately.
- Do not mix parser, runtime, and stdlib behavior changes in one patch unless absolutely necessary.
- When blocked, write the blocker into the owning module task file.

==================================================
21. First Release Slice Recommendation
==================================================

To avoid infinite scope, Release 0.1 should target:

- scripts and functions
- dense numeric arrays
- logical arrays
- chars/strings basic subset
- structs and cell arrays basic subset
- control flow
- multiple returns
- anonymous functions basic subset
- core builtins
- bytecode or native compile pipeline
- CLI-only workflow

Explicitly exclude from Release 0.1:

- advanced classdef completeness
- full graphics stack
- sparse matrix parity
- Simulink
- full toolbox support
- production-grade MEX compatibility
- GPU/distributed execution
