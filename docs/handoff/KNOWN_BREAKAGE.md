# Known Breakage

> Future AI: read `docs/handoff/FUTURE_ONLY.md` first. Use this file only when you need confirmed limitation/risk details for the area you are changing.

Current state:
- Full `cargo test --workspace` now passes in the Visual Studio developer environment, but a plain PowerShell shell still does not automatically have the MSVC toolchain on `PATH`.
- The repo now includes `scripts/cargo-msvc.cmd` and `scripts/cargo-msvc.ps1` as workarounds for normal shells, but plain `cargo test` still fails unless the MSVC toolchain environment has been activated first; for multi-flag commands, the `.cmd` wrapper is currently the reliable entry point.
- The parser now has a broader command-form baseline for statement-position calls and assignment right-hand sides, including dotted/path-like/range-like/flag-like raw arguments such as `pkg.sub`, `folder/file.txt`, `name-value`, `1:3`, `-force`, `./tmp/file`, `/tmp/file`, `\tmp\file`, `@helper`, and `../rel`, the current comma-separated command-argument subset like `strcmp alpha, alpha`, package-qualified command targets like `pkg.helper alpha`, contiguous operator-bearing raw arguments such as `name=value`, `alpha==beta`, `opt&&flag`, embedded quoted segments like `key="two words"` / `key='two words'`, grouped raw chunks like `value(1,2)`, `nested{1,2}`, `cell(1,[2,3])`, and `matrix([1,2;3,4])` that now keep inner comma/semicolon-separated balanced-delimiter content inside one command arg, raw-text joins across line continuations when the continuation does not introduce real separator trivia, and first assignment-RHS ambiguity preferences for arithmetic over bare signed numeric command args like `x = value -1`, signed grouped forms like `x = value -(1 + 2)` and `x = value +[1, 2]`, signed call-like forms like `x = value -sum([1, 2])`, signed field/index chains like `x = value -rhs.value` and `x = value +rhs.items(2)`, plus short bare division forms like `x = value /rhs`, `x = value \rhs`, `x = value ./rhs`, and `x = value .\rhs` when the target is a bare identifier, while still keeping richer signed raw text like `-1:3` or `+1ms` and longer raw path-like text such as `./tmp/file`, `/tmp/file`, or `\tmp\file` in command-form. But it is still intentionally narrow: only a first richer quoting/escaping slice now exists, and the full MATLAB command-vs-expression ambiguity surface is still not implemented.
- Package-qualified function calls and handles now work for dotted identifier chains, but that structure is still flattened to string names in semantic references instead of preserved as a richer dedicated semantic node.
- Resolver support now covers current directory, `private/`, package directories, and explicit `MATC_PATH` roots, and semantics now records final resolved-reference outcomes, but this is still a first filesystem-backed pass rather than full MATLAB path semantics.
- Global and persistent bindings now have shared identity semantics, but more MATLAB edge cases remain open, especially around cross-workspace interactions beyond the currently tested scenarios.
- The new HIR is intentionally structural: it preserves binding/capture/resolution detail, but it does not yet introduce control-flow blocks, phi-style merge modeling, or an execution-ready call/runtime abstraction.
- The optimizer crate now exists as a real first pass layer, but it is still intentionally early: it only does local HIR constant folding and branch/loop cleanup, with no CFG, dataflow, SSA, or interprocedural passes yet.
- The codegen crate now emits a real bytecode-style backend render with verifier support, the platform crate now exposes serialized bytecode artifacts plus dependency-aware bundles, and the execution crate now has a bytecode VM path that can execute both with parity against the current interpreter fixture suite. This backend is still intentionally early: bundles now carry explicit packaged module ids, but the manifest is still thin, compatibility negotiation is still limited to a simple format header, VM/debug tooling is still limited, there are no backend-only optimization passes, and there is no native backend integration.
- The interop crate now exposes a real workspace snapshot format and CLI export/inspection support, and that snapshot surface now round-trips the current first complex-scalar runtime values too, but this is still intentionally early: it is a MATC-specific snapshot format rather than MAT-file compatibility, it only covers the current runtime value surface, and it does not yet bridge FFI/MEX boundaries.
